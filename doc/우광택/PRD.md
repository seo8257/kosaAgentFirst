# PRD

## Product Requirements Document

## 프로젝트명: 추억의 노래방

## 1. 제품 개요

**추억의 노래방**은 4개의 개별 웹게임을 하나의 로비 화면에서 선택해 플레이할 수 있는 웹 기반 미니게임 플랫폼이다.

각 게임은 팀원별로 독립 개발되며, 모든 게임은 공통 CREDIT 시스템을 사용한다.

사용자는 로비에서 CREDIT을 충전하고, 게임을 시작할 때마다 CREDIT 100원을 사용한다.

---

## 2. 문제 정의

팀 프로젝트에서 여러 명이 각자 게임을 개발할 경우, 전체 서비스가 하나로 묶이지 않고 각각의 게임이 분리된 결과물처럼 보일 수 있다.

이를 해결하기 위해 하나의 공통 로비와 CREDIT 시스템을 제공한다.

공통 CREDIT 시스템을 통해 각 게임은 독립성을 유지하면서도 하나의 서비스 안에 포함된 것처럼 연결된다.

---

## 3. 제품 목표

### 3.1 핵심 목표

* 4개의 개별 웹게임을 하나의 서비스로 통합한다.
* 모든 게임이 하나의 CREDIT 데이터를 공유한다.
* 게임 시작 또는 재시작 시 CREDIT 100원을 차감한다.
* CREDIT이 부족하면 게임을 시작할 수 없게 한다.
* 팀원들이 각자의 게임을 독립적으로 개발할 수 있도록 한다.

---

### 3.2 비목표

본 프로젝트에서 다루지 않는 범위는 다음과 같다.

* 실제 결제 시스템
* 회원가입 및 로그인
* 서버 데이터베이스
* 온라인 랭킹 시스템
* 멀티플레이
* 모바일 앱 배포
* 상업적 서비스 운영

---

## 4. 사용자

### 4.1 주요 사용자

* 팀 프로젝트 결과물을 체험하는 사용자
* 발표를 듣는 청중
* 팀원 본인들
* 웹게임 제작을 실습하는 학습자

### 4.2 사용자 니즈

사용자는 다음 경험을 기대한다.

* 첫 화면에서 쉽게 게임을 고르고 싶다.
* 현재 CREDIT을 확인하고 싶다.
* CREDIT을 충전하고 싶다.
* 게임을 시작할 때 100원이 차감되는 것을 보고 싶다.
* CREDIT이 부족하면 왜 게임이 시작되지 않는지 알고 싶다.
* 게임이 끝난 뒤 다시 시작하거나 로비로 돌아가고 싶다.

## 5. 사용자 시나리오

### 시나리오 1: CREDIT 충전 후 게임 시작

1. 사용자가 첫 화면에 접속한다.
2. 현재 CREDIT이 0원으로 표시된다.
3. 사용자가 500원 충전 버튼을 누른다.
4. CREDIT이 500원으로 증가한다.
5. 사용자가 게임 1을 선택한다.
6. 게임 1 화면으로 이동한다.
7. 사용자가 게임 시작 버튼을 누른다.
8. CREDIT 100원이 차감된다.
9. CREDIT이 400원으로 표시된다.
10. 게임이 시작된다.

---

### 시나리오 2: 게임 오버 후 다시 시작

1. 사용자가 게임을 플레이한다.
2. 게임 오버가 발생한다.
3. 게임 오버 화면에 다시 시작 버튼이 표시된다.
4. 사용자가 다시 시작 버튼을 누른다.
5. CREDIT 100원이 차감된다.
6. 새 게임이 시작된다.

---

### 시나리오 3: CREDIT 부족

1. 현재 CREDIT이 0원이다.
2. 사용자가 게임 시작 버튼을 누른다.
3. 시스템은 CREDIT이 100원 미만임을 확인한다.
4. 게임은 시작되지 않는다.
5. 사용자에게 CREDIT 부족 메시지를 표시한다.

예시 메시지:

```text
CREDIT이 부족합니다. 로비에서 CREDIT을 충전해 주세요.
```

## 6. 기능 요구사항

## 6.1 로비 화면

### FR-001: 서비스 제목 표시

로비 화면에는 서비스 제목인 “추억의 노래방”을 표시한다.

### FR-002: CREDIT 표시

로비 화면에는 현재 CREDIT을 원화 단위로 표시한다.

예:

```text
CREDIT: 500원
```

### FR-003: CREDIT 충전

사용자는 로비 화면에서 CREDIT을 충전할 수 있다.

기본 충전 단위는 다음과 같다.

| 버튼       |  증가 금액 |
| -------- | -----: |
| 100원 충전  |  +100원 |
| 500원 충전  |  +500원 |
| 1000원 충전 | +1000원 |

### FR-004: 게임 선택

로비 화면에는 4개의 게임 선택 카드가 표시된다.

각 게임 카드에는 다음 정보가 포함된다.

* 게임명
* 간단한 설명
* 입장 버튼

### FR-005: 게임 화면 이동

사용자가 게임 입장 버튼을 누르면 해당 게임 화면으로 이동한다.

게임 입장 시 CREDIT은 차감하지 않는다.

---

## 6.2 공통 CREDIT 시스템

### FR-006: CREDIT 저장

CREDIT 값은 브라우저의 localStorage에 저장한다.

저장 키는 다음과 같이 통일한다.

```text
credit
```

### FR-007: CREDIT 조회

각 화면은 공통 함수를 통해 현재 CREDIT 값을 조회한다.

```js
getCredit()
```

### FR-008: CREDIT 충전

로비 화면은 공통 함수를 통해 CREDIT을 충전한다.

```js
addCredit(amount)
```

### FR-009: 게임 가능 여부 확인

각 게임은 게임 시작 전에 CREDIT이 100원 이상인지 확인해야 한다.

```js
canPlay()
```

### FR-010: CREDIT 차감

각 게임은 게임 시작 또는 재시작 시 다음 함수를 호출하여 CREDIT 100원을 차감한다.

```js
spendCredit()
```

`spendCredit()`은 CREDIT이 충분하면 100원을 차감하고 `true`를 반환한다.
CREDIT이 부족하면 차감하지 않고 `false`를 반환한다.

## 6.3 개별 게임 화면

### FR-011: 현재 CREDIT 표시

각 게임 화면은 현재 CREDIT을 표시한다.

### FR-012: 게임 시작 버튼

각 게임 화면에는 게임 시작 버튼이 있어야 한다.

게임 시작 버튼 클릭 시 반드시 `spendCredit()` 함수를 호출한다.

### FR-013: CREDIT 부족 처리

`spendCredit()`이 `false`를 반환하면 게임을 시작하지 않는다.

사용자에게 CREDIT 부족 메시지를 표시한다.

### FR-014: 게임 플레이

각 게임의 구체적인 플레이 방식은 팀원이 자유롭게 설계한다.

단, 게임은 웹 브라우저에서 실행 가능해야 한다.

### FR-015: 게임 오버

각 게임은 게임 종료 조건을 가져야 한다.

게임 오버 시 사용자에게 게임 종료 상태를 표시한다.

### FR-016: 다시 시작

각 게임은 다시 시작 기능을 제공한다.

다시 시작 시에도 반드시 `spendCredit()` 함수를 호출한다.

### FR-017: 로비로 돌아가기

각 게임 화면에는 로비로 돌아가기 버튼이 있어야 한다.

로비로 돌아갈 때 CREDIT은 유지된다.

## 7. 비기능 요구사항

### NFR-001: 독립 개발 가능성

각 게임은 독립적인 폴더 안에서 개발되어야 한다.

공통 CREDIT 모듈을 제외하면 게임 간 코드 의존성을 최소화한다.

### NFR-002: 단순한 실행 환경

프로젝트는 별도의 서버 없이 브라우저에서 실행 가능해야 한다.

### NFR-003: 데이터 지속성

CREDIT은 localStorage에 저장되어 브라우저를 닫았다 열어도 유지된다.

### NFR-004: 코드 일관성

모든 게임은 CREDIT 값을 직접 수정하지 않고 공통 CREDIT 함수를 사용해야 한다.

### NFR-005: 사용자 이해 가능성

CREDIT 충전, 차감, 부족 상태는 사용자가 쉽게 이해할 수 있도록 명확하게 표시되어야 한다.

## 8. 공통 CREDIT 모듈 명세

파일명:

```text
js/credit.js
```

### 상수

```js
const PLAY_COST = 100;
```

### 함수 명세

#### getCredit()

현재 CREDIT 값을 반환한다.

```js
getCredit() -> number
```

#### setCredit(amount)

CREDIT 값을 지정한 금액으로 저장한다.

```js
setCredit(amount)
```

#### addCredit(amount)

현재 CREDIT에 amount만큼 더한다.

```js
addCredit(amount)
```

#### canPlay()

현재 CREDIT이 100원 이상이면 true를 반환한다.

```js
canPlay() -> boolean
```

#### spendCredit()

CREDIT이 100원 이상이면 100원을 차감하고 true를 반환한다.
CREDIT이 부족하면 false를 반환한다.

```js
spendCredit() -> boolean
```

#### updateCreditDisplay(elementId)

지정한 화면 요소에 현재 CREDIT을 표시한다.

```js
updateCreditDisplay(elementId)
```

## 9. 권장 CREDIT 모듈 코드 초안

```js
const PLAY_COST = 100;
const CREDIT_KEY = "credit";

function getCredit() {
  return Number(localStorage.getItem(CREDIT_KEY) || 0);
}

function setCredit(amount) {
  localStorage.setItem(CREDIT_KEY, String(amount));
}

function addCredit(amount) {
  const current = getCredit();
  setCredit(current + amount);
}

function canPlay() {
  return getCredit() >= PLAY_COST;
}

function spendCredit() {
  if (!canPlay()) {
    return false;
  }

  setCredit(getCredit() - PLAY_COST);
  return true;
}

function updateCreditDisplay(elementId = "credit-display") {
  const element = document.getElementById(elementId);
  if (!element) return;

  element.textContent = `CREDIT: ${getCredit()}원`;
}
```

## 11. 각 게임 개발자가 지켜야 할 인터페이스

각 게임 개발자는 다음 규칙을 반드시 지킨다.

### 게임 시작 시

```js
function handleStartButtonClick() {
  if (spendCredit()) {
    updateCreditDisplay();
    startGame();
  } else {
    alert("CREDIT이 부족합니다. 로비에서 충전해 주세요.");
  }
}
```

### 다시 시작 시

```js
function handleRestartButtonClick() {
  if (spendCredit()) {
    updateCreditDisplay();
    restartGame();
  } else {
    alert("CREDIT이 부족합니다. 로비에서 충전해 주세요.");
  }
}
```

## 13. 성공 기준

프로젝트가 성공적으로 완성되었다고 판단하는 기준은 다음과 같다.

1. 로비 화면에서 CREDIT을 충전할 수 있다.
2. 로비 화면에서 4개의 게임으로 이동할 수 있다.
3. CREDIT 값이 모든 게임에서 동일하게 표시된다.
4. 게임 시작 시 CREDIT 100원이 차감된다.
5. 다시 시작 시에도 CREDIT 100원이 차감된다.
6. CREDIT이 100원 미만이면 게임이 시작되지 않는다.
7. 각 게임에서 로비로 돌아갈 수 있다.
8. 각 게임은 독립적으로 작동한다.
9. CREDIT 처리는 모든 게임에서 공통 모듈을 사용한다.

## 17. 최종 요약

**추억의 노래방**은 4개의 독립적인 웹게임을 하나의 로비와 공통 CREDIT 시스템으로 연결하는 팀 프로젝트이다.
각 팀원은 자신의 게임을 자유롭게 기획·개발하되, 게임 시작 및 재시작 시 CREDIT 100원을 차감하는 공통 규칙을 반드시 따른다.

이 프로젝트의 핵심은 “각자 만든 게임을 하나의 서비스처럼 보이게 만드는 최소한의 공통 시스템”이다.
